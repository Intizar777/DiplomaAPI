## Производство (Управление производством и технологиями)

### 1. ✅ OEE (Общая эффективность оборудования) 85%
**Формула:** Доступность × Производительность × Качество

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Доступность (Availability):** `DowntimeEvent` (простои оборудования)
  - Формула: (Плановое рабочее время - Время простоев) / Плановое рабочее время
  - Таблица: `downtime_events` (duration_minutes, started_at, ended_at)
  
- **Производительность (Performance):** `ProductionOutput` + `OrderSnapshot`
  - Формула: Фактический выпуск / Плановый выпуск
  - Таблица: `production_output` (quantity, production_date)
  
- **Качество (Quality):** `QualityResult`
  - Формула: Количество принятых партий / Всего партий
  - Таблица: `quality_results` (decision='accepted')

**План реализации:**
1. Создать модель `OEEMetric` с расчетом компонентов
2. Сервис: `oee_service.calculate_oee_by_line(production_line_id, period)`
3. Эндпоинт: `GET /api/v1/analytics/oee/summary` (по линиям и целевому значению 85%)

**Сложность:** Средняя (требует трех параллельных запросов)

### 2. ✅ Выход масла с тонны семян 42%
**Формула:** Масса масла / Масса семян × 100

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Входящие семена:** `BatchInput` (сырьевые материалы) - исходящее количество
- **Выходящееся масло:** `ProductionOutput` (готовая продукция) - выходное количество

**План реализации:**
1. Связать `BatchInput` с соответствующими `ProductionOutput` по order_id
2. Сервис: `yield_service.calculate_seed_to_oil_yield(period)`
3. Эндпоинт: `GET /api/v1/analytics/production/seed-to-oil-yield`

**Сложность:** Низкая (простое соотношение двух таблиц)

---

### 3. ✅ Производительность линии (т/сут)
**Формула:** Объём выпуска / Рабочее время

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Объём выпуска:** `ProductionOutput.quantity` (по production_date)
- **Рабочее время:** Из конфигурации линии или расчет (24h - простои из DowntimeEvent)

**План реализации:**
1. Агрегировать `ProductionOutput` по дату и production_line_id
2. Сервис: `productivity_service.get_line_productivity(line_id, period)`
3. Эндпоинт: `GET /api/v1/analytics/production/line-productivity`

**Сложность:** Низкая

---

### 4. ✅ Процент брака ≤ 1,5%
**Формула:** Объём брака / Объём выпуска × 100

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Объём брака:** `QualityResult` где decision ≠ 'accepted'
- **Объём выпуска:** `ProductionOutput.quantity`

**План реализации:**
1. Связать `QualityResult` с `ProductionOutput` по lot_number
2. Сервис: `quality_service.get_scrap_percent(period)`
3. Эндпоинт: `GET /api/v1/analytics/quality/scrap-percentage`

**Сложность:** Низкая (уже есть Quality Engineer Dashboard с аналогичной логикой)

### 5. ✅ Время простоев (ч) - Минимизация
**Формула:** Суммарное время незапланированных остановок

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Простои:** `DowntimeEvent.duration_minutes` (category = 'unplanned')

**План реализации:**
1. Агрегировать `DowntimeEvent` по production_line_id и дате

3. Сервис: `downtime_service.get_total_downtime(line_id, period)`
4. Эндпоинт: `GET /api/v1/analytics/production/downtime-summary`

**Сложность:** Низкая (уже есть 2-я волна дашборда с Pareto по простоям)

## Продажи (Управление продажами и маркетингом)

### 11. ✅ Объём продаж (т, млн руб.)
**Формула:** Количество реализованной продукции по каналам сбыта

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Количество (т):** `SaleRecord.quantity` по channel
- **Сумма (млн руб.):** `SaleRecord.amount` по channel

**План реализации:**
1. GROUP BY channel, GROUP BY product, GROUP BY region
2. Сервис: `sales_service.get_volume_by_dimension(dimension, period)`
3. Эндпоинт: `GET /api/v1/analytics/sales/volume-by-channel`
4. **Примечание:** Уже есть Finance Manager Dashboard с sales-breakdown

**Сложность:** Низкая (уже реализовано)

---

### 12. ✅ OTIF (≥ 95%)
**Формула:** Заказы в срок и в полном объёме / Всего заказов × 100

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Заказы в срок:** `OrderSnapshot` где actual_delivery_date ≤ planned_delivery_date
- **В полном объёме:** `OrderSnapshot` где delivered_quantity ≥ ordered_quantity
- **Всего заказов:** COUNT(*) в OrderSnapshot

**План реализации:**
1. Связать OrderSnapshot с SaleRecord по order_id
2. Проверить условия: delivery_date ≤ planned_date AND quantity_delivered = quantity_ordered
3. Сервис: `otif_service.get_otif_percentage(period, filters)`
4. Эндпоинт: `GET /api/v1/analytics/sales/otif`

**Сложность:** Средняя (требует проверки двух условий)

---

### 13. ✅ Эффективность промо-акций (> 1.0)
**Формула:** Прирост продаж в период акции / Затраты на акцию

**Статус:** ✅ **РЕАЛИЗУЕМО**

**Источники данных:**
- **Промо-акции:** `PromoCampaign` (start_date, end_date, budget)
- **Продажи в период:** `SaleRecord` (sale_date между start_date и end_date)
- **Базовая продажа (контроль):** SaleRecord в аналогичный период прошлого года

**План реализации:**
1. Для каждой `PromoCampaign`:
   - Рассчитать продажи за период акции
   - Рассчитать базовый уровень продаж (год назад)
   - Прирост = Продажи за период - Базовый уровень
   - Эффективность = Прирост / Budget
2. Сервис: `promo_service.calculate_promo_effectiveness(campaign_id, period)`
3. Эндпоинт: `GET /api/v1/analytics/sales/promo-effectiveness`

**Сложность:** Средняя (требует расчета контрольной группы)
